Organization

What is the structure of TinyKV?

  1. Each machine has a Server with a certain object that implemented Storage interface to store Key/Value pairs.
    • The Storage interface only requires to provide Read and Write APIs. They way they implement these methods decided the service can be provided by this Server.
    • If that Storage is simply using badger.DB that directly writes data, the Server only provides a standalone database service.
    • If that Storage is RaftStorage that completes these methods by interacting with Raft groups, it can provide a distributed fault-tolerence database.
  2. Inside RaftStorage, there is a RaftStore that controls all peers. Each peer is a peer in a raft group.
    • When multi-raft is enabled, there may be several peers in the same RaftStore that manages different shards.
  3. Overall, a RaftStorage is responsible for an entire key space. All peers in the same key space, though may be in charge of incontinuous shards, must communicate with clients through this RaftStorage.
    • If there exists another key space, e.g. another database, a separate RaftStorage must be created.

How to execute a command from client?

  1. The following is a diagram of execution flow.
  2. Clients give commands through the APIs provided by Server. Those APIs will execute Read or Write provided by Storage.
  3. RaftStorage will send a message of raft_cmdpb.Request to RaftStore through RaftRouter. The request contains information of regionId, peerId for router to locate the destination.
  4. The RaftRouter will send the request to the RaftWorker of RaftStore. When the RaftWorker received messages, it will initiate a peerMsgHandler for each message one by one.
    • Notably that RaftWorker will start the next peerMsgHandler until the last one is finished, instead of creating goroutines to process them in parallel.
    • Raft peers will only receive commands and messages from RaftWorker. Hence, only one Raft thread in the entire RaftStore processes messages, rather than handling them in parallel.
  5. peerMsgHandle will ask RaftGroup to handle the message by Step.
  6. After all read messages are handled, all peers that have received messages are asked to handle raft.Ready one be one. In this process, applied commands will be executed, persisted states will be written in to PeerStorage.

Time Representation

How does Raft represent time?

  1. Instead of using a TrueTime API to represent time like Spanner, TinyKV uses a logical time to control workflows.
  3. All peers in the same store should register on tickDriver so that it will know which peers it need to send message when events are triggered.
  4. Each store has a store ticker while each peer has a peer ticker. The store ticker is the driver of the entire store, it represents the logical time of this machine. The peer tickers are simply for schedule peer events.
  5. When the store ticker received from time.Tick(baseTickInterval),
    • A MsgTypeTick will be sent to all known peers. Each peer will check whether their schudele is up.
    • Then, tick is increased by 111 and tickDriver checks the schedule of the store. If a schedule is up, the driver will send a message of MsgTypeStoreTick to the store.

What events are scheduled in peers?

  1. When the tickDrive of RaftStore sends a MsgTypeTick to RaftWorker, the peerMsgHandler would check several schedules.
  2. When PeerTickRaft is up, the Raft node ticker is moved forward.
  3. When PeerTickRaftLogGC is up and existing applied entries are more then the limit, it will propose a Raft command with administrator request to compact Raft log to the applied index when this request is made.
    • The compaction is only executed when this request is committed.
    • The Raft node and peer storage modify their state according to the compact index and term. Then a garbage collection task is sent to the raftLogGCWorker which will remove compacted entries from RaftDB.
  4. When PeerTickSchedulerHeartbeat is up, a SchedulerRegionHeartbeatTask is sent to schedulerWorker.

How does transactions represent timestamps?

  1. The placement driver is also responsible for assigning timestamps to clients for transaction events.
    • The placement driver is another Raft group powered by etcd. Only the leader can assign timestamps. When the physical timestamp is changed, the leader will save a timestamp that is seconds later then the new physical timestamp.
    • When a new leader is elected, it will read from etcd files to sync its timestamp with the last saved timestamp. The new leader needs to wait until the time of the read timestamp comes before it can assign any new timestamp.
    • We can induct that the timestamp assign by the PD group is unique and monotonously increasing. Therefore, the TiKV did not take the TrueTime API in Spanner.
  2. A transaction timestamp is an uint64 that consists of two parts:
    • The high bits are the physical time with unit of millisecond. The lower bits are the logical time to distinguish the events in the same millisecond.
    • To prevent logical time becoming too large, it is resetted to zero every time the physical time is updated.
  3. The server will check whether need to update the physical timestamp every 50 ms50\ ms50 ms, instead of every millisecond.
    • The physical timestamp will sync with current time when it is not updated for over 150 ms150\ ms150 ms.
    • When used over half of the logical timestamp, the physical timestamp will add 1 ms1\ ms1 ms and reset the logical timestamp.
